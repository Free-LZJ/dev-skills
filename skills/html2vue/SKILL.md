# Html2vue

## 概览

将用户提供的 HTML 源码作为 UI 原型，把页面开发为 Vue 实现时，必须以视觉一致性为最高约束。允许按项目设计模式拆分组件、接入状态、接口、路由和 i18n，但最终渲染效果必须尽量复用原 HTML 的结构、class、CSS 语义和视觉细节。

## 使用边界

使用本 skill 的条件：

- 用户提供 HTML 源代码、HTML 文件、静态页面原型、可运行的 HTML/CSS/JS 原型。
- 用户要求将 HTML 原型开发为 Vue、Vue 2、Vue 3、SFC、Composition API、Options API、TSX/JSX 或项目既有 Vue 写法。
- 用户强调高保真、像素级、视觉一致、还原 UI、布局一致、配色一致、class 保留、DOM 对齐。

不使用本 skill 的情况：

- 用户只提供截图、Figma、口头描述或设计稿链接，没有 HTML 源码。此时先要求用户提供 HTML 源码，或改用截图/设计稿还原类流程。
- 用户要求重新设计页面、优化视觉风格、换一套交互体验。此时按普通前端设计/开发流程处理。
- 用户只要求抽取数据逻辑、接口联调或修 bug，且不涉及 HTML 原型还原。

## 输入要求

开始开发前，必须先获得或定位以下材料：

- 原始 HTML 源码或 HTML 文件路径。
- 原型依赖的 CSS、JS、图片、字体、图标资源，或其可访问路径。
- 目标 Vue 项目的技术栈：Vue 2/Vue 3、SFC/JSX、TypeScript、UI 库、CSS 方案、构建工具。
- 目标页面入口、路由、组件目录、i18n 资源位置和现有编码规范。
- 原型目标视口，默认至少覆盖桌面 `1440x900` 和移动 `375x812`。

如果资源缺失，先标记缺口并保持尺寸占位一致。不得擅自替换成不同风格的图片、字体、图标或颜色。

## 工作流

1. 解析原型

- 阅读 HTML、CSS 和资源引用，梳理页面模块、DOM 层级、class 命名、布局方式、颜色、字体、字号、行高、间距、阴影、圆角、边框、图片尺寸和交互状态。
- 给页面划分模块边界，例如头部、筛选区、列表、卡片、弹窗、空状态、底部区域。
- 记录每个模块的关键视觉约束，尤其是宽高、display、grid/flex 规则、padding、margin、gap、定位、z-index、overflow。

2. 制定 Vue 映射

- 优先复用原 HTML 的 DOM 结构和 class 命名。
- 组件拆分按项目设计模式执行，但拆分后每个组件渲染出的 DOM 外观必须与原模块一致。
- 将重复结构抽为组件或列表渲染时，保持最终 DOM 层级、class、顺序、语义标签和 CSS 选择器稳定。
- 将静态内容替换为数据绑定时，保持默认示例数据与原型视觉长度接近，避免文字长度导致布局变化。
- 按项目约定处理 i18n；存在 i18n 机制时，用户可见文字必须进入 locales，不得在模板或脚本展示逻辑中硬编码。

3. 搬迁样式

- 优先迁移原 CSS 变量、class、选择器和局部结构。
- 不随意调整 spacing、font、color、line-height、border、radius、shadow、opacity、DOM 层级。
- 必须修改样式时，先说明原因，例如适配组件作用域、消除全局污染、接入现有设计系统或修复响应式问题。
- 避免为了“更 Vue”而重命名 class；如确需重命名，保留原 class 作为兼容层或在交付说明中列出映射关系。
- 对 `position: absolute`、固定高度、滚动容器、图片裁切、文字省略、表格列宽等敏感区域做专项核对。

4. 接入行为

- 仅在不影响视觉的前提下接入状态、事件、接口、表单校验、路由和权限逻辑。
- 保持原型已有 hover、active、disabled、loading、empty、modal、dropdown、tooltip、tab 等状态的视觉表现。
- 如原型没有覆盖某状态，按项目既有设计系统补齐，并在交付说明中标明新增状态。

5. 模块级比对

- 按模块逐一比对原型和 Vue 渲染结果。
- 每个模块至少检查 DOM 层级、关键 class、宽高、间距、颜色、字号、行高、对齐方式、图片比例、交互状态。
- 发现差异时，优先修 Vue/CSS 实现；只有在原型明显不可运行或与项目强约束冲突时，才调整映射方案。

6. 浏览器视觉检查

- 交付前必须运行目标页面。
- 使用浏览器打开原 HTML 和 Vue 页面，在相同视口下截图检查。
- 至少检查桌面和移动两个视口；如页面有复杂响应式断点，增加对应断点。
- 对关键模块执行 DOM 检查，确认 class、层级和布局盒模型与原型保持一致。
- 若环境支持 `browser-use`、Playwright 或项目现有 e2e 工具，优先用工具截图和检查；否则手动浏览器检查并在最终回复中说明。

## 高保真原则

- 视觉一致优先于代码审美。
- 结构复用优先于重写布局。
- class 保留优先于重命名。
- CSS 搬迁优先于重新设计。
- 组件拆分服务于工程维护，不改变最终视觉 DOM。
- 响应式适配允许补充规则，但不得破坏原型主视口效果。
- 所有偏离原 HTML 的地方都必须有明确工程理由。

## 禁止事项

- 禁止随意改变 DOM 层级、标签类型、class 命名或 CSS 选择器。
- 禁止随意改动颜色、字号、行高、间距、圆角、阴影、边框、宽高和布局方式。
- 禁止把原型中的复杂布局简化成“看起来差不多”的卡片或通用栅格。
- 禁止因为组件化而改变最终渲染结构。
- 禁止用项目默认样式覆盖原型关键视觉。
- 禁止未经说明替换图片、字体、图标、背景或资源比例。
- 禁止只完成代码不做浏览器视觉检查。

## 交付清单

最终回复必须包含：

- 原型来源：HTML 文件或源码位置。
- Vue 实现位置：页面、组件、样式、i18n 资源。
- 保真策略：哪些 DOM/class/CSS 被复用，哪些地方因工程原因做了调整。
- 比对结果：桌面和移动视口的视觉检查结论。
- 未决差异：资源缺失、字体不可用、原型脚本不可运行、项目约束导致的差异。
- 验证命令：启动、构建、测试、截图或浏览器检查命令。

## 差异处理

遇到视觉差异时，按以下顺序处理：

1. 检查资源是否加载一致，包括图片、字体、图标、CSS reset、浏览器默认样式。
2. 检查盒模型差异，包括 `box-sizing`、display、flex/grid、width、height、padding、margin、gap、border。
3. 检查文字差异，包括 font-family、font-size、font-weight、line-height、letter-spacing、white-space。
4. 检查定位和层叠，包括 position、z-index、transform、overflow、scrollbar、sticky/fixed。
5. 检查 Vue 条件渲染、列表渲染、slot、组件库默认样式是否改变 DOM 或 CSS。
6. 记录仍无法消除的差异，并给出原因和影响范围。
