# Requirements Delivery

本 skill 用于把需求文档转成可持续交付的工作流，并用多文件工作集降低上下文腐烂。

如果用户只提供工单号，默认到 `doc/<工单号>/` 下查找文档；如果用户指定了路径，以用户路径为准。新建或更新配套文档时，应放在同一目录。

## 核心契约

默认使用多文件工作集：

| 文件 | 角色 | 默认读取 |
|------|------|----------|
| `当前上下文.md` | 新会话恢复入口，保存当前有效结论、已失效结论、活跃 Task、下一步和必须复验项 | 必读 |
| `系统分析.md` | 当前设计与分析基线 | 按活跃 Task 精读 |
| `开发清单.md` | Task 拆分、阶段化状态、当前审查门禁、最新结论和证据索引 | 必读总览和活跃 Task |
| `测试计划.md` | 验证矩阵、测试场景和执行记录 | 按活跃 Task 精读 |
| `过程记录.md` | 变更、Bug、审计明细、复盘和历史过程 | 默认不读 |

`过程记录.md` 是给人看和审计用的历史归档。除非用户要求历史、审计、追责、复盘，或活跃文档明确引用某条历史记录，否则不要读取它，也不要让它覆盖当前有效结论。

## 按需加载

`SKILL.md` 只保留入口规则。按当前任务阶段读取细节：

| 当前问题 | 读取 |
|------|------|
| 新建/迁移文档结构、当前上下文、过程记录边界 | `references/document-set.md` |
| `/analyze`、`/sync`、新会话恢复、实现门禁、验证顺序 | `references/workflow.md` |
| Task 阶段化状态、开发清单模板、审查门禁、审计明细迁移 | `references/task-review-audit.md` |
| superpowers 协同、brainstorm/plan/debug/review 使用边界 | `references/superpowers-coordination.md` |
| 需要一个已落地案例 | `references/peil-practice-only-case.md` |

不要一次性读取所有 references。只打开当前决策需要的文件。

## 启动顺序

1. 先读取仓库规则，包括 `AGENTS.md`。
2. 如果继续已有需求，先读 `当前上下文.md`。
3. 再读 `开发清单.md` 总览和当前活跃 Task。
4. 再读相关 `系统分析.md` 和 `测试计划.md` 章节。
5. 只有在需要历史、审计、追责、复盘，或活跃文档引用了历史记录时，才读 `过程记录.md`。

改代码前保持门禁顺序：

1. 需求或设计变化时，先更新系统分析/设计。
2. 更新 `当前上下文.md`。
3. 更新受影响 Task 的状态、验收口径、审查门禁和验证链接。
4. 除非用户明确绕过门禁，否则只有在分析和 Task 状态与目标一致后再实现。

## 活跃文档规则

- `当前上下文.md` 应足够短，便于每次新会话读取，通常控制在 100-150 行。
- `系统分析.md` 保留当前设计和必要约束，不承载长篇执行历史。
- `开发清单.md` 保留 Task 结构、阶段状态、当前审查门禁、最新结果、证据链接和过程记录索引。
- `测试计划.md` 保留验证用例和执行记录。
- `过程记录.md` 保留长篇历史、审计明细、Bug 记录、多轮审查和追责材料。

结论变化时，先更新所有受影响的活跃文件，再继续实现。旧结论应标记为已失效，不能让新旧结论并存且不说明优先级。

## 强制执行规则

- 区分文档假设、代码真实状态、已验证的模型/配置真实状态。
- Task 宁可拆细，也不要混合多个不相关目标。
- 分析变化后不能继续执行旧 Task，必须先同步。
- 涉及数据和 SQL 时，必须写清查询策略：来源、过滤、关联、聚合、批量、排序、分页和 N+1 规避。
- 复杂查询、跨表聚合、多来源合并分页、权限过滤下推、统计报表类查询，优先数据库下推。
- SQL 涉及 `UNION ALL`、`CASE`、`COALESCE`、`NULL`、占位列或跨类型比较时，必须审查渲染 SQL 和 PostgreSQL 类型兼容性。
- 公开分页接口优先复用平台分页结构，如 `ListPageResult<T>`；除非系统分析已证明需要额外业务字段，否则不要新增只包裹分页字段的 VO。
- 不要自动提交。只有用户明确要求时，才暂存或提交。

## 审查与审计边界

按上下文价值拆分审查内容：

| 内容 | 默认位置 |
|------|----------|
| 当前审查门禁 / 当前检查项 | `开发清单.md` |
| 当前审查结论摘要 | `开发清单.md` |
| 详细审计过程 | `过程记录.md` |
| 多轮审查记录 | `过程记录.md` |
| 逃逸缺陷、复盘、责任归因 | `过程记录.md` |
| 测试命令和执行结果 | `测试计划.md` |

当审查内容超过当前门禁和一句结论时，把明细迁入 `过程记录.md`；`开发清单.md` 只保留当前门禁、最新结论、验证指针和过程记录链接。

## Skill 协同

只在任务需要时加载额外 skill：

- `$epaas-router`：宽泛 ePaaS 后端问题分流。
- `$epaas-server`：Hook、事件、MQ、定时任务或服务链路。
- `$datarecord`：查询、计数、EQL、关联路径和数据更新规则。
- `$sql-template`：SQL 编写、渲染、模板、动态 SQL 或 PostgreSQL 兼容性审查。
- `superpowers`：用户明确要求，或 workflow reference 判断当前阶段确实需要方法层支持。

当 requirements-delivery 与 superpowers 同时启用时，requirements-delivery 负责权威工作文档和 Task/status 流；superpowers 只作为方法层。
