# 工作流

处理 `/analyze`、`/sync`、实现回合、验证回合和新会话恢复时，读取本文件。

## 命令

| 命令 | 用途 |
|------|------|
| `/analyze <需求来源>` | 从需求来源启动新的系统分析 |
| `/analyze <需求来源> --from <现有工作文档>` | 从现有工作文档继续，补充新需求 |
| `/analyze <需求来源> --task <Task编号>` | 针对特定 Task 补充分析 |
| `/analyze <需求来源> --brainstorm` | 用户显式要求时，正式分析前按内置 `brainstorming` 子技能收敛需求 |
| `/fast <Task编号> <目标>` 或 `--fast` | fast 模式：最快完成代码，不走子技能，不记录过程文档，只更新开发清单状态 |
| `/debug <Task编号> <描述>` | 按内置 systematic-debugging 子技能排查根因，并同步工作文档 |
| `/verify <Task编号>` | 按内置 verification-before-completion 子技能做完成前验证 |
| `/sync` | 同步当前代码状态到权威工作文档 |
| `/sync --task <Task编号>` | 同步特定 Task 状态 |
| `/bug <Task编号> <描述>` | 在 `过程记录.md` 记录 Bug，活跃文档只保留当前结论、待办和索引 |
| `/change <Task编号> <描述>` | 在 `过程记录.md` 记录变更，活跃文档同步当前有效结论和 Task 状态 |
| `/tasks` | 查看当前开发清单 Task 状态 |
| `/task <Task编号>` | 查看特定 Task 详情 |

## 启动

1. 读取 `AGENTS.md`、目标文档和相关模块，再编辑代码。
2. 如果继续已有需求，先读 `当前上下文.md`。
3. 读取 `开发清单.md` 总览和当前活跃 Task。
4. 读取相关 `系统分析.md` 和 `测试计划.md` 章节。
5. 只有在需要历史、审计、追责，或活跃文档引用了某条历史记录时，才读取 `过程记录.md`。

## 分析

产出或修订系统分析时，应覆盖：

- 业务背景和目标
- 当前实现状态
- 需求与代码/模型真实状态的差异
- 受影响模块、数据模型、Hook、Controller、SQL、外部集成
- 足以指导编码的实现设计
- 查询路径、写入路径、分支条件、兼容处理、校验点
- 相关 helper/API/skill 的使用预期
- 风险、假设和验证计划

系统分析是可编辑的。用户调整目标、范围、术语、优先级、兼容约束或 Task 边界时，先同步下游文档，再继续实现。

## Task 转换

把权威系统分析拆成小 Task，每个 Task 至少包含：

- 目标
- 受影响代码路径
- 验收信号
- 当前阶段状态

可分离时，按查询变更、服务变更、Hook 变更、Controller 暴露、文档同步、验证、兼容处理拆分。

## 设计变更回合

默认顺序：

1. 更新相关系统分析/设计章节。
2. 更新 `当前上下文.md`，写清当前有效结论、必要的已失效结论、影响 Task 和下一门禁。
3. 更新受影响 Task 的状态、验收口径、验证链接和过程记录索引。
4. 如需保留历史，把长篇明细追加到 `过程记录.md`。
5. 告知用户变更点并等待确认，除非用户明确绕过门禁。
6. 用户确认后再改代码，并再次同步状态。

## 实现回合

默认顺序：

1. 如果实现改变设计，先更新相关系统分析/设计章节。
2. 更新 `当前上下文.md`，保证下一会话可恢复。
3. 更新受影响 Task 状态、验收口径、验证链接和过程记录索引。
4. 只实现当前 Task。
5. 做本地局部复核；涉及 SQL 时检查类型对齐。
6. 运行最小有意义的编译或测试。
7. 如发现缺陷，先本地修复，再标记完成。
8. 验证通过后，更新 Task 阶段状态、`测试计划.md` 执行证据和 `当前上下文.md`；长篇审查或 Bug 过程写入 `过程记录.md`。

## Fast 模式

用户明确说 `fast模式`、`--fast`、`快速模式` 或“最快完成代码”时启用。

执行规则：

1. 只读取完成代码所需的最小上下文：仓库规则、`开发清单.md` 活跃 Task、必要代码。
2. 不调用 `brainstorming`、`systematic-debugging`、`verification-before-completion` 或其他内置方法子技能。
3. 不走 TDD 流程，不要求先写失败测试。
4. 不更新 `当前上下文.md`、`系统分析.md`、`测试计划.md`、`过程记录.md`。
5. 直接实现代码，运行最小有意义验证；为节省时间无法验证时，在最终回复说明。
6. 只更新 `开发清单.md` 中相关 Task 的阶段状态、当前状态和必要证据。

退出 fast 模式后，恢复默认 requirements-delivery 工作流。

## 新会话恢复模板

推荐用户这样发起继续任务：

```text
使用 requirements-delivery skill 继续处理这个需求。

当前上下文：D:\project\...\doc\<工单号>\当前上下文.md
系统分析：D:\project\...\doc\<工单号>\系统分析.md
开发清单：D:\project\...\doc\<工单号>\开发清单.md
测试计划：D:\project\...\doc\<工单号>\测试计划.md
过程记录：D:\project\...\doc\<工单号>\过程记录.md（默认不读取，除非需要审计历史）
本次目标：继续完成 Task X
流程要求：
1. 先读取当前上下文
2. 再修改系统分析
3. 同步当前上下文、Task 清单和状态
4. 我确认后再改代码
5. 每完成一个 Task，必须先本地复核；有问题直接修复后再更新状态
```

## 验证

先验证最小有意义范围：

1. 编译受影响模块。
2. 运行定向测试。
3. 只有在必要时运行更大范围验证。

代码变更和选定验证步骤都完成前，不要写完成结论。

不要自动提交。只有用户明确要求时才提交。
